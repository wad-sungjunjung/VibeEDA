## 1. Vibe EDA 한눈에 보기

Vibe EDA 는 **자연어로 SQL/Python 을 생성·수정하고, 분석 결과를 리포트로 뽑아내는** 로컬 전용 AI 네이티브 노트북입니다. Jupyter 의 상위 버전이라 생각하면 됩니다.

- **Vibe Chat**: 한 셀 단위로 "이 쿼리에 시도별 group by 추가해줘" 같은 자연어 수정
- **에이전트 모드**: 여러 셀에 걸친 분석 작업을 자동으로 수행
- **리포팅**: 분석을 마친 셀을 골라 Markdown 리포트로 저장
- **Claude/Gemini 이중 지원**: 목적과 비용에 따라 모델 자유 전환
- **노트북 저장 형태**: `.ipynb` (Jupyter 호환)

핵심 저장 위치는 모두 `~/vibe-notebooks/` 하위입니다.

```
~/vibe-notebooks/
├── {분석 제목}.ipynb      # 노트북 본체 (셀·채팅·에이전트 히스토리 포함)
├── .vibe_config.json      # 폴더 메타데이터
└── reports/
    ├── {YYYYMMDD_HHMMSS}_{slug}.md
    └── {report_id}_images/*.png
```

## 3. 분석 흐름 개요

```
① 분석 주제 입력
   └─ 예: "광고 시도 유입 변화 확인"
        ↓
② 마트 선택 (직접 고르거나 추천)
        ↓
③ 셀 생성 & 코드 작성 (SQL / Python / MD)
   └─ 직접 쓰기 · 바이브 챗 · 에이전트 중 선택
        ↓
④ 실행 & 결과 확인
   └─ 차트, 표, 메모로 기록
        ↓
⑤ 리포트 생성
   └─ 분석 내용을 Markdown 으로 저장, 이미지 임베드
```

각 단계는 아래 섹션에서 자세히 다룹니다.

## 4. 마트 설정 — 가장 먼저 제대로 해두어야 할 것

**왜 중요한가:** 선택된 마트만 LLM 의 SQL 컨텍스트에 들어갑니다. 무관한 마트까지 넣으면 프롬프트가 길어지고, 엉뚱한 조인을 제안하는 원인이 됩니다.

### 4.1 마트 선택 방법

- **분석 시작 시**: 노트북을 만든 직후 마트 모달이 뜨면, 분석 주제를 입력 → `마트 추천` 버튼 → LLM 이 3–5개를 제안
- **나중에 변경**: 상단 메타 헤더에 있는 마트 태그 영역 클릭 → 추가·제거

### 4.2 추천 원리

`POST /v1/marts/recommend` 엔드포인트가 등록된 마트 카탈로그 전체를 LLM 에게 보여주고, 분석 주제와 매칭도가 높은 순으로 후보를 돌려받습니다. 이미 알고 있는 마트가 있다면 수동으로 추가 가능.

### 4.3 마트 메타 구조

각 마트는 `key`, `description`, `columns[].(name,type,desc,categories)` 를 가집니다. 특히 `categories` 는 "이 컬럼에 들어올 수 있는 값 허용 리스트"로, 바이브 챗이 WHERE 절에 정확히 그 값을 쓰도록 시스템 프롬프트에 주입됩니다. 카테고리에 없는 값으로 필터링하면 결과가 비기 때문입니다.

### 4.4 자주 하는 실수

- ❌ 쓰지도 않는 마트를 잔뜩 선택해두기 → 비용·정확도 모두 나빠짐
- ❌ 카테고리 없는 컬럼에 가정으로 WHERE 걸기 → 모델이 존재하지 않는 값 추측 가능
- ✅ 실제 분석에 쓰는 3–5개 정도 유지

## 5. 셀 편집 기본기

### 5.1 셀 타입

| 타입 | 용도 | 실행 결과 |
|---|---|---|
| **SQL** | Snowflake 쿼리 | DataFrame 테이블 |
| **Python** | 분석/차트/가공 | 표, Plotly 차트, stdout |
| **Markdown** | 설명·메모 | 렌더된 텍스트 |

셀 왼쪽 배지(SQL/PY/MD) 를 클릭하면 타입이 순환됩니다.

### 5.2 셀 이름 (snake_case 식별자)

**왜 이름이 중요한가:** SQL 셀 결과는 `{셀이름}` 이름의 Python 변수로 커널에 주입됩니다.

```sql
-- SQL 셀 이름: shop_revenue
select shop_id, sum(revenue) from ... group by 1
```

```python
# Python 셀 (아래쪽)
import plotly.express as px
fig = px.bar(shop_revenue, x='shop_id', y='sum(revenue)')
```

- ✨ 버튼: 코드를 LLM 에 보여주고 snake_case 이름 자동 추천 (가장 저렴한 모델 사용 — Gemini Flash Lite 우선)
- 직접 입력 시 영문 소문자+숫자+언더스코어만 허용 (자동 검증)

### 5.3 레이아웃

셀 우상단의 3개 버튼으로 전환:

- `□` 단일: 탭(입력/출력/메모) 전환
- `▬` 좌우 분할: 입력·출력 동시에
- `≡` 상하 분할: 입력·출력 수직 배치

구분선을 드래그해 비율 조정. 셀 하단의 얇은 핸들로 전체 패널 높이도 조절 가능(노트북별 저장).

### 5.4 전체화면

- **진입**: 본문 더블클릭 혹은 Maximize 버튼
- **해제**: ESC
- **전환**: 우측 셀 네비게이션에서 다른 셀 클릭 → 상하 슬라이드로 부드럽게 교체

좌측 분석 목록과 우측 네비게이션은 유지되어, 전체화면 중에도 셀 이동이 자연스럽습니다.

### 5.5 단축키

| 단축키 | 동작 |
|---|---|
| `Ctrl+Enter` | 활성 셀 실행 |
| `Ctrl+L` | 바이브 챗으로 포커스 |
| `Ctrl+B` | 활성 셀 아래 동일 타입 셀 추가 |
| `Ctrl+G` | 에이전트 모드 토글 |
| `Tab / Shift+Tab` | 들여쓰기 / 내어쓰기 |
| `Ctrl+/` | 라인 주석 토글 |

좌측 사이드바 → 설정 → `단축키` 에서 전체 목록 참고.

## 6. 바이브 챗 (단일 셀 편집용)

활성 셀 하단의 채팅 박스에 자연어로 요청을 쓰면 해당 셀의 코드만 수정·재작성합니다.

### 6.1 예시 프롬프트

| 상황 | 요청 |
|---|---|
| SQL | "시도별 group by 추가하고 매출 내림차순 정렬" |
| SQL | "최근 7일로 기간 좁혀줘" |
| Python | "pie 차트로 바꾸고 상위 10개만" |
| Python | "결과 표를 query_1 기준으로 left join 해서 비율 컬럼 추가" |
| MD | "위 차트를 본 인사이트 3줄로 정리" |

### 6.2 컨텍스트 주입 정책

- **SQL 셀**: 선택된 마트의 스키마 + 카테고리 허용값 + `code_style.py` 의 스타일 가이드
- **Python 셀**: 직전 SQL 셀 이름, 커널에 살아있는 DataFrame 요약(`df.head()`·shape 정보) 주입
- **MD 셀**: 위쪽 셀 요약 일부만 간단히

### 6.3 모델 선택

- 채팅 박스 좌하단 드롭다운으로 즉시 교체
- 비용 순: `gemini-2.5-flash-lite` < `gemini-2.5-flash` < `claude-haiku-4-5` < `gemini-2.5-pro` < `claude-sonnet-4-6` < `claude-opus-4-7` / `gemini-3.1-pro-preview`
- 정확도가 필요하면 Pro/Opus, 간단한 수정은 Flash Lite 로

## 7. 에이전트 모드 (멀티셀 자동화)

**언제 쓰는가:** "매출 변화 분석해줘" 처럼 마트 탐색 → SQL → 차트 → 해석까지 한 번에 필요할 때.

### 7.1 실행

- 우측 하단 FAB(Floating Action Button) 클릭 또는 `Ctrl+G`
- 오른쪽에 에이전트 채팅 패널 등장
- 질문 입력 → 스트리밍으로 진행 상황이 실시간 표시됨

### 7.2 도구(툴)들

에이전트가 호출 가능한 10개 툴(요약):

| 툴 | 역할 |
|---|---|
| `get_mart_schema` | 선택된 마트의 컬럼·타입·카테고리 조회 |
| `preview_mart` | 마트 상위 몇 행 미리보기 |
| `profile_mart` | 컬럼별 null 비율/분포 요약 |
| `create_sql_cell` / `create_python_cell` / `create_markdown_cell` | 새 셀 삽입 |
| `update_cell` | 기존 셀 코드 수정 |
| `execute_cell` | 셀 실행 + 결과 수신 |
| `write_memo` | 셀 메모에 인사이트 기록 (가드로 강제) |
| `ask_user` | 진행 중 사용자에게 되묻기 |

차트 결과는 **PNG 이미지로 tool_result 에 주입**되어, 에이전트가 다음 단계에서 차트를 "눈으로 보고" 판단합니다.

### 7.3 안전 장치

- **메모 강제 가드**: 분석 셀을 만들고 실행한 뒤 반드시 `write_memo` 로 인사이트를 적게 함 → 리포트 품질 향상
- **반복 호출 가드**: 동일 툴을 N회 이상 연속 호출하면 중단하고 질문으로 전환
- **`ask_user` 이벤트**: 불확실하면 선택지를 주고 멈춤 — 사용자가 답하면 이어서 진행

### 7.4 모델 기본값

- 기본: **Claude Opus 4.7** — tool use 안정성이 압도적으로 좋음
- 실험: **Gemini 3.1 Pro** 도 동일 툴 셋으로 돌아감(코드베이스: `gemini_agent_service.py` 가 `claude_agent.py` 의 `_execute_tool` 을 공유)
- 고비용이므로 간단한 탐색은 바이브 챗을 권장

### 7.5 예시 시나리오

> "최근 한 달 시도별 매출 변화 보여줘"

에이전트가 수행하는 일련의 흐름:

1. `get_mart_schema(mart_revenue)` → 필요한 컬럼 확인
2. `preview_mart` → 실제 데이터 느낌 파악
3. `create_sql_cell` → 시도별·일자별 집계 쿼리 생성
4. `execute_cell` → 결과 확인
5. `create_python_cell` → 시계열 라인 차트 코드 생성
6. `execute_cell` → 차트 렌더 (PNG 이미지 tool_result 로 회수)
7. `write_memo` → "최근 일주일 서울 시도가 ~~ 증가" 식 인사이트 기록

자세한 파이프라인 구조는 `docs/vibe-eda-agent-pipeline.md`.

## 8. 실행 & 결과 다루기

### 8.1 실행

- 헤더 호버 시 나타나는 ▶ 버튼 혹은 `Ctrl+Enter`
- 상단 "모두 실행" 버튼: SQL/Python 셀을 순서대로 일괄 실행 (Markdown 은 제외)

### 8.2 커널 상태

- 노트북별 **독립 Python 커널 네임스페이스** 유지
- SQL 결과는 `{셀이름}` 변수로 주입됨
- 커널 리셋이 필요하면 `DELETE /v1/kernel/{notebook_id}` (추후 UI 추가 예정)

### 8.3 출력 복사

- **표**: 우상단 복사 버튼 → TSV 로 클립보드 (Google Sheets·Excel 바로 붙여넣기)
- **차트**: 복사 버튼 → 2x 해상도 PNG 로 클립보드 (Slack·문서 붙여넣기)

### 8.4 표 고정 열

표 좌측의 얇은 세로 바를 드래그하면 해당 경계까지의 컬럼이 고정됩니다. Google Sheets 의 freeze panes 와 동일한 UX. 가장 왼쪽으로 끌면 고정 해제.

### 8.5 차트 크기

Plotly `layout.width/height` 를 존중합니다. 셀 패널은 차트 크기에 맞춰 자동 확장되지만, 수동으로 리사이즈한 값이 있으면 그걸 우선. PNG 는 600x400 기본으로 kaleido 가 자동 렌더(대부분 차트에 자동 적용).

## 9. 리포팅 파이프라인

### 9.1 리포트 생성 흐름

1. 상단 `리포팅` 버튼
2. 리포트 모달에서:
   - **포함할 셀 선택** (체크박스)
   - **리포트 목표** 입력 (예: "A/B 테스트 효과 요약")
   - **모델 선택** (기본 Claude Opus 4.7)
3. `생성 시작` → SSE 로 Markdown 실시간 스트리밍
4. 완료되면 `~/vibe-notebooks/reports/{YYYYMMDD_HHMMSS}_{slug}.md` 로 저장
5. 좌측 사이드바 `reports` 섹션에서 언제든 재열람

### 9.2 Evidence 빌드

선택된 셀마다 아래 재료가 LLM 에 들어갑니다:

- 셀 제목 · 타입 · 코드
- 결과 표(최대 N rows) 또는 stdout 일부
- 차트는 `{report_id}_images/` 에 PNG 로 저장되고 상대 경로로 Markdown 에 임베드

### 9.3 결과물 형식

- YAML frontmatter(제목·생성시각·모델·포함 셀 id 등) + 본문 Markdown
- 이미지 경로는 상대 경로(`./{report_id}_images/chart_1.png`) → 리포트 폴더째 이동해도 동작

### 9.4 Tip

- **좋은 메모 → 좋은 리포트**: 각 셀에 에이전트/분석가가 남긴 메모가 리포트 내러티브의 뼈대가 됩니다.
- 숫자 강조가 필요한 분석은 Opus 4.7 을 권장 — 수치 정합성이 가장 안정적

자세한 내부 구조는 `docs/vibe-eda-reporting-pipeline.md`.

## 10. 분석 내용 정리 팁

> "좋은 분석" 이 되려면 셀 배치와 메모가 중요합니다.

- **한 셀 = 한 질문**: SQL 셀에 10개 지표를 욱여넣지 말고, 질문 단위로 쪼개 주세요.
- **명확한 이름**: ✨ 버튼으로 빠르게 snake_case 이름 — 변수 공유 시 가독성 UP
- **Markdown 셀로 흐름 쓰기**: "여기까지는 유입, 이제부터 전환" 같은 세션 구분
- **각 분석 셀 끝에 메모**: 인사이트 한 줄 — 에이전트가 가드로 강제해주기도 함
- **차트는 레이블 명확히**: Plotly `layout.xaxis.title`, `yaxis.title` 필수. 리포트에 그대로 실립니다.
- **리포트 전에 재실행**: `모두 실행` 으로 순서 꼬임 없음을 확인

## 11. 문제 해결 (FAQ)

| 증상 | 원인 · 대응 |
|---|---|
| SQL 셀이 "연결 실패" | 좌측 사이드바 → 연결 관리 → 재연결 / externalbrowser SSO 브라우저 팝업 차단 여부 확인 |
| 바이브 챗이 엉뚱한 컬럼 사용 | 선택된 마트가 실제 분석과 맞는지 점검 · 카테고리 허용값 업데이트 |
| 에이전트가 무한 루프 같음 | 내부 가드가 중단시키지만 수동 중지 가능 · 더 구체적인 질문으로 재시도 |
| Plotly 차트가 PNG 로 안 나옴 | `pip install kaleido` 누락 가능 — 백엔드 requirements.txt 재설치 |
| 다크모드 전환 시 차트 배경이 흰색 | 재실행 필요 — 저장된 PNG 는 만들 당시 테마를 유지하므로 셀 실행을 한 번 다시 돌리면 현재 테마로 재렌더 |
| 리포트 이미지가 깨짐 | `{report_id}_images/` 폴더가 함께 있는지 확인 — 리포트와 이미지 폴더는 한 쌍으로 이동 |
