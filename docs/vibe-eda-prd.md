## 1. Executive Summary

### 1.1 한 줄 소개
> **Jupyter Notebook + Claude Desktop 하이브리드** — 분석가가 자연어 채팅으로 SQL/Python 코드를 생성·수정하며 EDA를 수행하고, 최종적으로 리포트 초안까지 자동 생성하는 AI 네이티브 분석 도구.

### 1.2 비전
"바이브 코딩"을 넘어 **"바이브 EDA"**: 도메인 맥락(마트 메타데이터, 비즈니스 규칙)을 LLM에 주입하여 분석가가 코드 문법이 아닌 **분석적 사고**에 집중할 수 있게 한다.

### 1.3 타겟 사용자
- **주요**: 사내 광고 플랫폼 데이터 분석가 (Snowflake + Python 사용)
- **부차**: PM, 마케터 중 SQL을 일부 다루는 파워유저

## 2. 문제 정의

### 2.1 현재 상황의 페인포인트

1. **도메인 지식의 파편화**
   - 마트 스키마, 비즈니스 규칙(지역 상한, 단가 정책 등)이 분석가 개개인의 머릿속에 존재
   - 신규 입사자는 "어떤 마트에 뭐가 있는지" 파악하는 데 수주 소요

2. **반복적 EDA 작업**
   - 매번 유사한 집계 쿼리를 작성 (시도별, 상품별, 기간별 등)
   - 익숙한 시각화 코드를 처음부터 타이핑

3. **리포트 작성 부담**
   - 분석이 끝나도 리포트로 옮기는 데 별도 시간 투입
   - SQL, 차트, 인사이트를 수동으로 조합

4. **LLM 도구의 한계**
   - ChatGPT/Claude에 붙여넣으면 도메인 맥락이 없어서 "generic"한 답변
   - Copilot은 코드 보조는 되지만 데이터 분석 흐름과는 단절

### 2.2 사용자 Job-to-be-done
> "나는 분석 주제를 정하면, 어떤 데이터를 써야 할지 가이드받고, 코드를 일일이 외우지 않고도 데이터를 탐색하고, 발견한 인사이트를 깔끔한 리포트로 남기고 싶다."

## 3. 목표 & 성공 지표

### 3.1 비즈니스 목표
- 분석가 1인당 **EDA 소요 시간 40% 단축**
- 신규 입사자의 **도메인 온보딩 기간 50% 감소**
- 리포트 작성 시간 **70% 감소**

### 3.2 제품 지표 (Key Metrics)
| 지표 | 정의 | 목표 (3개월) |
|---|---|---|
| WAU | 주간 활성 사용자 | 사내 분석가 20명 중 15명 |
| 세션당 셀 수 | 한 번 접속 시 평균 셀 생성 개수 | 10+ |
| 바이브 사용률 | 전체 셀 중 채팅으로 수정된 셀 비율 | 60%+ |
| 에이전트 셀 생성율 | 에이전트가 자동 생성한 셀 비율 | 15%+ |
| 리포트 완성률 | 노트북 완료 시 리포트까지 생성하는 비율 | 50%+ |

### 3.3 비-목표 (Non-Goals, MVP에서 하지 않는 것)
- 대시보드 공유/퍼블리싱
- 다중 사용자 실시간 협업 (동시 편집)
- 모바일 지원
- 데이터 소스 직접 연결 (MVP는 Snowflake 전용)
- ETL, 데이터 파이프라인 기능

## 4. 핵심 사용자 시나리오

### Scenario A: 신규 분석 시작
1. 분석가 "하우"가 **신규 분석** 시작
2. **분석 주제** 입력: "지역별 광고 매출 분석"
3. **분석 내용** 상세 기술: "최근 7일간 시도/시군구..."
4. 분석 내용 기반 **마트 추천** (상위 3개) 확인
5. 추천 중 `ad_sales_mart`를 선택 → 우측 "사용할 마트"로 이동
6. **SQL 셀** 자동 생성 → 기본 쿼리 로드
7. 하단 **바이브 채팅** 입력: "시도별로 group by 해줘"
8. 쿼리가 자동 수정되고 실행 → 결과 테이블 표시
9. "pie 차트로 시각화" → **Python 셀** 수동 추가 후 바이브로 지시
10. 모든 셀 체크 → **리포팅** 버튼 클릭 → Markdown 리포트 자동 생성

### Scenario B: 기존 분석 재활용
1. 좌측 히스토리에서 "상품별 CTR 추이" 선택
2. 과거 노트북 불러옴
3. 셀의 **대화 이력**에서 이전 시점 코드로 **롤백**
4. 기간만 바꿔서 재실행

### Scenario C: 에이전트 모드 탐색적 분석
1. 1차 쿼리 실행 후 인사이트 포착
2. 우측 하단 **FAB 클릭** → 에이전트 모드 ON
3. 하단 고정 채팅창에 "강남구 세부 분석해줘"
4. 에이전트가 관련 SQL 셀을 **자동 생성** 후 실행
5. "보러가기" 클릭 → 생성된 셀로 스크롤
6. 생성된 셀에서 추가로 바이브 수정

### Scenario D: 히스토리 정리
1. 히스토리가 20개 이상 쌓임
2. **폴더 추가** 버튼 클릭 → "Q4 분석" 생성
3. 관련 히스토리의 `⋯` → 이동 → "Q4 분석" 선택
4. 자주 쓰는 분석은 **복제** → 새 분석의 출발점으로 활용

## 5. 기능 요구사항

### 5.1 필수 (MVP, Must-have)

#### F1. 분석 메타데이터 관리
- F1.1 분석 주제 1줄 입력 (필수)
- F1.2 분석 내용 상세 입력 (권장, 상세할수록 추천 정확도 ↑)
- F1.3 상단 메타 헤더 접기/펼치기 (`▾`/`▸`)
- F1.4 접힌 상태에서 주제 + 리포팅 버튼 노출

#### F2. 마트 관리 (다중 선택)
- F2.1 전체 마트 카탈로그 노출 (좌측 풀 영역)
- F2.2 분석 내용 기반 **점수 기반 추천** (상위 노출)
- F2.3 마트명/설명/키워드/컬럼명 **전문 검색**
- F2.4 클릭 또는 `+` 버튼으로 "사용할 마트"에 추가
- F2.5 선택된 마트의 `X` 버튼으로 제거
- F2.6 각 마트의 `Info` 아이콘으로 **컬럼 스키마 + 비즈니스 규칙** 펼치기
- F2.7 2개 이상 선택 시 **JOIN 관계 표시** (`mart1 ⋈ mart2`)

#### F3. 셀 타입 & 편집
- F3.1 3가지 타입 지원: SQL (Snowflake), Python (Plotly), Markdown
- F3.2 **타입 배지 클릭으로 순환 전환** (SQL → Python → MD → SQL)
  - 타입 전환 시 **코드/탭 상태 유지** (파괴적 변경 없음)
- F3.3 셀 이름 인라인 편집 (예: `daily_sales`)
  - SQL 셀의 이름은 후속 Python 셀에서 **DataFrame 변수명으로 참조**
- F3.4 탭 전환: "입력" / "출력"
  - 타입 무관 일관된 라벨
- F3.5 셀 실행 버튼 (▶, 호버 시 노출)
- F3.6 셀 삭제 버튼 (호버 시 노출)
- F3.7 하단 **셀 추가 바 (고정)** — SQL / Python / Markdown
- F3.8 새 셀은 **현재 활성 셀 바로 아래**에 삽입

#### F4. 바이브 채팅 (셀 단위)
- F4.1 활성 셀에만 채팅창 노출 (비활성 셀은 숨김)
- F4.2 자연어 입력 → LLM이 코드/쿼리 수정
- F4.3 Enter = 전송, Shift+Enter = 줄바꿈
- F4.4 대화 이력 저장 (코드 스냅샷 포함)
- F4.5 대화 개수 표시 ("대화 3")
- F4.6 각 대화 카드 클릭 시 **해당 시점 코드로 롤백**
- F4.7 현재 코드 = 어떤 대화 시점인지 "현재" 배지 표시

#### F5. 에이전트 모드
- F5.1 우측 하단 **FAB (Floating Action Button)** 으로 토글
- F5.2 활성 시 하단 중앙에 **Claude 스타일 채팅 패널** 노출
- F5.3 노트북 전체 맥락 기반 질의응답
- F5.4 에이전트가 **새 셀을 자동 생성** 가능
- F5.5 생성된 셀은 "에이전트" 배지 표시
- F5.6 응답에 "셀 N개 생성 · 보러가기" 링크 → 해당 셀로 스크롤
- F5.7 추천 프롬프트 칩 (빈 대화 시): "강남구 세부 분석해줘" 등
- F5.8 우측 사이드바 하단에 **에이전트 이력 누적 표시**

#### F6. 히스토리 & 폴더
- F6.1 좌측 사이드바에 히스토리 목록
- F6.2 폴더 생성 (`FolderPlus` 아이콘)
- F6.3 폴더 펼치기/접기
- F6.4 히스토리 컨텍스트 메뉴 (`⋯`):
  - **이동** → 하위 메뉴에서 폴더 선택 (현재 위치는 ✓ 표시)
  - **복제** → 원본 바로 아래에 "(복제)" 접미어로 사본 생성
  - **삭제** (빨간색, 파괴적 액션 강조)
- F6.5 폴더 삭제 시 안의 히스토리는 **루트로 자동 이동** (유실 방지)

#### F7. 리포팅
- F7.1 상단 헤더 우측 **"리포팅"** 버튼
- F7.2 모달에서 **포함할 셀 선택** (체크박스)
- F7.3 Markdown 리포트 자동 생성:
  - 제목 (분석 주제)
  - 메타데이터 (날짜, 사용 마트, 셀 수)
  - 분석 배경 (분석 내용)
  - 각 셀의 코드 + 출력 (테이블은 샘플 3행)
  - Markdown 셀은 본문 그대로 삽입
- F7.4 복사 버튼 → 클립보드로 전체 내용
- F7.5 실행되지 않은 셀은 선택 불가 (체크박스 disabled)

### 5.2 권장 (Should-have)

#### S1. 우측 셀 네비게이션

## 6. 비기능 요구사항 (Non-Functional Requirements)

### 6.1 성능
- 셀 1개 실행: **2초 이내** 응답 (Mock 기준)
- LLM 응답 대기 시 **로딩 인디케이터** 필수
- 셀 50개까지 스크롤 랙 없음

### 6.2 보안
- 사내 SSO (Okta 등)로 인증
- Snowflake 접근은 **사용자 권한 상속** (대리 접근 금지)
- 분석 내용은 **개인 워크스페이스 기본**, 공유는 명시적 액션

### 6.3 안정성
- 바이브 채팅 실패 시 **원본 코드 보존**
- 에이전트 생성 셀 실패 시 에러 토스트만, 기존 셀 영향 없음
- 히스토리 자동 저장 (30초 간격)

### 6.4 관찰성 (Observability)
- LLM 호출 횟수 / 응답 시간 로깅
- 사용자당 세션 길이, 셀 생성 이벤트 추적
- 에러는 Sentry 등으로 자동 수집

## 7. 기술 스택 (실제 구현)

### 7.1 Frontend (구현 완료)
- **React 18 + Vite + TypeScript**
- **Tailwind CSS v3** (커스텀 디자인 토큰 — `tailwind.config.ts`)
- **Lucide React** 아이콘
- **Zustand** 상태 관리 (`src/store/useAppStore.ts`, `modelStore.ts`, `connectionStore.ts`)
- **CodeMirror 6** 코드 편집기 (`src/components/cells/CodeEditor.tsx`)
- **react-markdown** + remark-gfm (마크다운 렌더링)
- **Plotly.js** 차트 출력

### 7.2 Backend (구현 완료)
- **Python FastAPI** (`backend/`)
- **Claude API (Anthropic)** — `claude-opus-4-7` / `sonnet-4-6` / `haiku-4-5`
- **Gemini API (Google)** — `gemini-2.5-flash` / `pro` 등, 런타임 스위치 지원
- **Snowflake Python Connector** (externalbrowser SSO)
- **In-process Python exec** (Jupyter Kernel Gateway 대신 노트북별 namespace 직접 관리)
- **MCP SDK** (Claude Code 연동용 별도 프로세스)

### 7.3 Infrastructure (로컬 전용)
- **로컬 전용 앱** — Kubernetes, PostgreSQL, Redis, S3 미사용
- **노트북 저장**: `~/vibe-notebooks/*.ipynb` (사용자 홈)
- **폴더 메타**: `~/vibe-notebooks/.vibe_config.json`
- **채팅/에이전트 히스토리**: 동일 `.ipynb` 의 `metadata.vibe` 하위에 임베드
- **향후 확장용 미사용 코드**: `backend/app/database.py`, `models.py` (SQLAlchemy 모델 정의만 존재)
- **Dockerfile / infra/**: 존재하나 현재는 로컬 구동이 기본 경로

| M2: LLM 연동 | ✅ 완료 | Vibe 채팅 + Agent 모드 (Claude + Gemini 이중) |
| M3: 실행 엔진 | ✅ 완료 | Snowflake 쿼리 + in-process Python 커널 + `.ipynb` 저장 |
| M4: Claude Code MCP 연동 | ✅ 완료 | MCP 서버 (`backend/app/api/mcp_server.py`) |
| M5: 에이전트 모드 고도화 | 🟡 진행 중 | ask_user, 셀 메모, 재계획 UX 개선 |
| M6: 리포팅 완성 | 예정 | Markdown 외 포맷 (HTML/PDF) 확장 |
| M7: 파일럿 | 예정 | 5명 내부 테스트 + 피드백 반영 |

### 9.2 Gating Criteria (출시 기준)
- [ ] 핵심 시나리오 A~C를 error 없이 완주
- [ ] p95 응답 시간 < 5초
- [ ] 파일럿 NPS > 40
- [ ] Security review 통과
- [ ] 문서화 완료 (사용자 가이드, 운영 가이드)

## 10. 리스크 & 대응

| 리스크 | 심각도 | 대응 |
|---|---|---|
| LLM 응답 품질 편차 | High | 프롬프트 엔지니어링 전담, 도메인 지식 주입 강화, eval set 구축 |
| Snowflake 쿼리 비용 폭증 | Medium | 쿼리당 cost 추정 + 경고, 행 수 LIMIT 자동 주입 |
| 사용자 데이터 유출 | High | LLM 전송 전 PII 마스킹, 감사 로그, 전송 옵트아웃 |
| 복잡한 분석에서 AI 한계 | Medium | 에이전트 모드에서 "모르겠음" 명시, 수동 전환 옵션 |
| 성능 저하 (셀 100개+) | Low | 가상 스크롤, 실행 결과 지연 로드 |
